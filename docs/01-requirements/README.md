# 要件定義

## 1. 背景・課題

Expoアプリを新しく作るたびに、初期設定や共通機能を一から実装すると開発開始までに時間がかかる。

一方、複数アプリで同じ共通部品を直接共有すると、共通部品の変更によって複数のアプリが同時に壊れるおそれがある。

そのため、実用的な共通スターターを各アプリへコピーし、コピー後はアプリごとに独立して管理する方式を採用する。

## 2. 目的

Expo・React Native・TypeScriptを使ったアプリ開発で、毎回発生する初期設定や共通機能の実装を減らすため、フロントエンドからバックエンド接続まで含む共通スターターを作成する。

完成したスターターは各アプリへコピーして利用し、コピー後はアプリごとに独立して管理する。これにより、既存アプリへの破壊的な影響を防ぎながら、短期間で複数のアプリを開発できる状態を目指す。

まずはFinance Appで実際に必要となった構成や機能を基に育て、将来的には他の開発者も利用できる公開テンプレート兼ポートフォリオとする。

**品質を落とさず、アプリを高速に量産できる土台を作る。**

## 3. 利用者

- 主な利用者はリポジトリ所有者本人とする
- Expoを使った新しいアプリを開発するときに利用する
- 将来的には、GitHub上で他の開発者も利用できる公開テンプレートを目指す

## 4. 対象アプリ

- Android・Webで動作するExpoアプリ
- 小規模から中規模のアプリ
- 初期版ではiOSの動作保証を対象外とする

## 5. 利用方法

- GitHubからスターター一式をコピーして新しいアプリを作成する
- コピー後のコードは各アプリで独立して管理する
- スターター側の変更をコピー後のアプリへ自動反映しない
- 必要な改善だけを各アプリへ手動で取り込む

## 6. 初期版に含めるもの

代表的な既存スターターで一般的に用意されている内容を参考にし、次の機能と開発基盤を初期版へ含める。

- 基本ディレクトリ構成
- Expo Routerによる画面遷移
- テーマ
- 基本UI部品
- UIラッパー
- 認証
- ログイン状態管理
- API通信
- Repository
- フォーム
- 入力検証
- ローディング表示
- エラー処理
- 環境変数管理
- 多言語化の土台
- ESLint
- Formatter
- 単体テストの設定
- 画面テストの設定
- GitHub Actions
- Development・Staging・Productionの環境分離
- ログイン画面
- アカウント画面
- 新しい画面や部品を追加する手順書

既存スターターの内容を無条件に取り込むのではなく、本スターターの目的と技術構成に合うものを選定し、不要な機能や過剰な依存関係は追加しない。

## 7. 初期版に含めないもの

- iOSの動作保証
- Firebase・AWSなど、Supabase以外のバックエンドへの対応
- Finance App固有の家計簿機能
- アプリ公開・ストア申請の自動化
- 課金
- 広告
- プッシュ通知
- アカウント削除・プロフィールの物理削除（V2で対応）

## 8. 技術・動作環境

- Expo
- React Native
- TypeScript
- Supabase
- Android・Web対応
- Node.jsの安定版
- GitHubによるソースコードと設計資料の管理

具体的なライブラリ、構成、バージョンは内部設計で決定する。

## 9. 非機能要件

### セキュリティ

- 秘密鍵や個人情報をGitへ保存しない
- Development・Staging・Productionを安全に分離する
- Publicリポジトリとして安全に公開できる状態を保つ

### 品質

- Android・Webの両方で正常に動作する
- 自動テスト、Lint、型チェックを実行できる
- エラー発生時に原因を確認できる表示またはログを残す
- 採用する依存ライブラリの保守状況と安全性を確認する
- 不要な依存ライブラリを増やさない

### 保守性・拡張性

- 機能を追加・変更しやすい構成にする
- コピー後のアプリを独立して保守できる
- 新しい画面や部品の追加方法を文書化する

### 使いやすさ

- 初めて利用する人でもREADMEを読めば起動できる
- アクセシビリティを考慮する
- 一般的なスマートフォンとPCで快適に操作できる

## 10. 完成条件

初期版は、次の条件をすべて満たした時点で完成とする。

- READMEの手順だけでセットアップして起動できる
- Android・Webの両方で基本画面が動作する
- Development・Staging・Productionを切り替えられる
- Supabaseへ接続できる
- ログイン、ログアウト、認証状態の保持が動作する
- 画面遷移、フォーム、入力検証、エラー表示が動作する
- テーマ、多言語化、基本UI部品を利用できる
- テスト、Lint、型チェックが成功する
- GitHub Actionsで品質確認を自動実行できる
- 新しい画面や部品の追加手順が文書化されている
- 秘密情報を含まず、Publicリポジトリとして安全に公開できる
- コピーして別アプリとして独立運用できる

## 11. 前提・制約

- Finance Appで実際に試しながら改善する
- Publicリポジトリに置ける内容だけを含める
- 秘密情報や実際の利用者データを含めない
- コピー後のアプリへスターターの変更を自動反映しない
- 初期版ではAndroid・Webのみ動作保証する
- 外部サービスの利用料金が発生する構成では、無料枠や代替手段を明記する
- 採用技術や依存ライブラリは、保守状況と安全性を確認して選ぶ

## 12. 用語

| 用語 | 意味 |
|---|---|
| スターター | 新しいアプリ開発の出発点としてコピーする雛形 |
| コピー後のアプリ | スターターから作成し、独立して管理する各アプリ |
| UI部品 | ボタンや入力欄など、画面で再利用する部品 |
| UIラッパー | 利用ライブラリの違いを吸収する、自作の共通UI部品 |
| Repository | 画面からデータの取得・保存処理を切り離す仕組み |
| Development | 開発・動作確認用の環境 |
| Staging | 本番公開前の確認環境 |
| Production | 実際に利用する本番環境 |

## 13. 未決事項

現時点ではなし。パスワード・表示名の制約およびプロフィール削除をV2へ延期する方針は、外部設計・内部設計で確定済みとする。
