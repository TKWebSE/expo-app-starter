# 内部設計・DB

コード構成、責務分担、データと外部サービスの接続方法を定義します。

## 設計方針

- V1は複数アプリへコピーしやすい、薄くて堅い土台にする
- 一般的な構成とライブラリを採用し、独自ルールを増やさない
- 実際に利用して判明した問題はV2で改善する
- 特定アプリの業務ロジックやテーブルはスターターへ含めない
- 独自バックエンド、複雑なDI、汎用的すぎる基底クラスは作らない

## ディレクトリ構成

```text
src/
├─ app/          # Expo Routerの画面と画面遷移
├─ features/     # 認証、プロフィール、設定などの機能
├─ components/   # アプリ全体で使う共通UI
├─ services/     # Supabaseなど外部サービスとの接続
├─ stores/       # アプリ全体で共有する状態
├─ hooks/        # 共通Hooks
├─ theme/        # 色、文字、余白など
├─ i18n/         # 翻訳リソースと多言語化設定
├─ config/       # 環境変数とアプリ設定
├─ utils/        # 汎用処理
├─ types/        # 共通の型
└─ test/         # テスト支援
```

機能固有のコードは `features/<feature-name>/` にまとめます。

```text
features/auth/
├─ components/
├─ hooks/
├─ repositories/
├─ schemas/
├─ stores/
├─ types/
└─ utils/
```

すべての機能で全フォルダを作る必要はありません。必要になったものだけ追加します。

## レイヤーと依存関係

依存方向は次に統一します。

```text
画面 → 機能Hook / Store → Repository → Supabaseなどの外部サービス
```

- 画面からSupabaseを直接呼び出さない
- DBや外部APIの操作はRepositoryに集約する
- 共通部品から特定機能へ依存しない
- `features` 同士の直接依存は避け、必要な共有処理は共通層へ切り出す
- 循環依存を作らない

## UIラッパー

画面ではUIライブラリの部品を直接多用せず、共通UIラッパーを経由します。

- 色、余白、文字サイズ、角丸などをテーマへ集約する
- ボタン、入力欄、ダイアログ、Snackbarなどの見た目と挙動を統一する
- アプリ固有の複合部品は各 `features` に置く
- ラッパーは必要な機能だけ公開し、UIライブラリの全APIを再現しない

## 状態管理

| 状態                               | 管理方法            |
| ---------------------------------- | ------------------- |
| サーバーから取得するデータ         | TanStack Query      |
| フォーム入力                       | React Hook Form     |
| 入力検証                           | Zod                 |
| 複数画面で共有するクライアント状態 | Zustand             |
| 画面内だけで完結する状態           | Reactのローカル状態 |

サーバーデータをZustandへ重複保存しません。テーマなど端末固有の設定はローカルストレージへ保存します。

## 認証とセッション

- Supabase Authのメールアドレスとパスワード認証を使用する
- 起動時に保存済みセッションを復元する
- 認証状態の監視結果をアプリ全体で共有する
- 未認証ユーザーは認証画面、認証済みユーザーはアプリ画面へ振り分ける
- Service Role Keyをフロントへ置かない
- V1では複雑な権限ロールを導入しない
- パスワードは8文字以上64文字以下とし、文字種の組み合わせは強制しない
- パスワード確認はクライアント側で一致を検証し、パスワード自体はDBへ保存しない

## Repositoryと外部サービス境界

- Repositoryは画面やUIに依存しない
- Supabase固有のレスポンスをそのまま上位層へ渡さない
- DB行はアプリで扱う型へ変換して返す
- 外部サービスの例外は共通エラー形式へ変換する
- Repositoryのインターフェースをテスト時に差し替えられる構成にする

## DB設計

スターターのV1では `profiles` テーブルだけを用意します。アプリ固有テーブルはコピー先で追加します。

| カラム         | 型            | 制約・用途                                |
| -------------- | ------------- | ----------------------------------------- |
| `id`           | `uuid`        | 主キー。 `auth.users.id` と同じ値         |
| `display_name` | `text`        | 必須。前後の空白を除いて1〜50文字の表示名 |
| `created_at`   | `timestamptz` | 作成日時                                  |
| `updated_at`   | `timestamptz` | 更新日時                                  |

- `auth.users` と `profiles` は1対1とする
- 日時はDBでUTCとして保存し、表示時に端末のタイムゾーンへ変換する
- V1ではプロフィールの物理削除機能を提供しない
- DB変更はSupabase MigrationのSQLとしてGit管理する
- 新しいアプリ固有テーブルには原則として `user_id`、`created_at`、`updated_at` を設ける

## RLSと権限

- 公開スキーマの全テーブルでRLSを有効にする
- 利用者は自分の行だけ参照・追加・更新できるようにする
- V1ではDELETEポリシーを作成せず、Repositoryにも削除処理を実装しない
- ポリシーでは `auth.uid()` と所有者IDを比較する
- RLSを無効化して動作させる運用は禁止する
- 管理権限が必要になった場合はフロントではなく信頼できるサーバー側で扱う

## 環境と秘密情報

- Development、Staging、ProductionでSupabaseプロジェクトを分離する
- 環境変数の名前と用途を `.env.example` に記載する
- フロントには公開可能なSupabase URLとAnon Keyだけを置く
- Service Role Keyや外部サービスの秘密鍵をリポジトリおよびアプリへ含めない
- 環境変数の不足は起動時に検出し、開発者が原因を判断できるエラーにする

## エラー処理とログ

共通エラーは少なくとも次を区別します。

- 入力エラー
- 認証エラー
- 通信エラー
- 権限エラー
- データ未検出
- 想定外エラー

利用者向けメッセージと開発ログを分離し、内部情報や秘密情報を画面へ表示しません。画面上の通知方法は外部設計の共通フィードバック規則に従います。

## 多言語化

- V1の初期表示言語は日本語とする
- 画面文言をコードへ直書きせず、翻訳リソースから取得できる土台を用意する
- V1ですべての英訳を必須にはしない

## テストしやすさ

- Repositoryをモックへ差し替えられるようにする
- HookとStoreはUIから分離してテストする
- Zodスキーマは境界値と不正値をテストする
- RLSは別ユーザーのデータを操作できないことを確認する
- 実装詳細ではなく利用者から見た挙動を優先してテストする

## V2で見直す項目

実際にスターターからアプリを作った後、次を材料に改善します。

- コピー時に毎回修正が必要だった箇所
- 重複実装が発生した箇所
- UIラッパーで不足または過剰だったAPI
- 状態管理の責務が分かりにくかった箇所
- テストや障害調査が難しかった箇所
- セキュリティ、性能、アクセシビリティ上の問題
- アカウント削除と `profiles` の削除方法

V2は推測で機能を増やすのではなく、実利用で確認できた問題を優先して改修します。
